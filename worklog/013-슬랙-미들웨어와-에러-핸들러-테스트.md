## 2. BoltRequest mock — 가벼운 fake 클래스로 충분

slack-bolt 의 `BoltRequest` 를 진짜로 만들면 의존성이 줄줄이 따라와요. 미들웨어는 `req.body` 와
`req.context` 만 만지니까 두 속성만 흉내내는 fake 로 충분해요.

```python
class FakeContext(dict):
    """req.context.user_id 와 req.context["service"] 양쪽 접근을 모두 지원."""
    def __init__(self, user_id=None, channel_id=None):
        super().__init__()
        self.user_id = user_id
        self.channel_id = channel_id

class FakeRequest:
    def __init__(self, body, user_id=None, channel_id=None):
        self.body = body
        self.context = FakeContext(user_id=user_id, channel_id=channel_id)
```

`FakeContext` 가 `dict` 를 상속하니까 `ctx["service"] = ...` 도 되고, 명시 속성 `user_id`/`channel_id`
접근도 돼요. 이런 작은 어댑터 패턴이 미들웨어 테스트의 키예요.

## 3. log_event_middleware — body 모양 분류기

이 미들웨어의 핵심은 **여러 body 모양을 통일된 (event, type) 쌍으로 분류** 하는 거예요.

| body 모양            | event           | type            |
| -------------------- | --------------- | --------------- |
| {"command": "/제출"} | "/제출"         | "command"       |
| {"type": "view_submission", "view": {"callback_id": ...}} | callback_id | "view_submission" |
| {"type": "block_actions", "actions": [{"action_id": ...}]} | action_id | "block_actions" |
| {"event": {"type": "app_mention"}} | "app_mention" | "event" |
| 기타 (알 수 없음)    | "unknown"       | "unknown"       |

각 분류를 한 케이스씩 박아두고, message/reaction 두 분기는 **명시적으로 로깅을 우회** 해야 해서
별도 테스트로 뺐어요. (handle_message 와 handle_reaction_added 가 자체적으로 로깅하기 때문)

```python
async def test_log_event_middleware_skips_message_event(mocker):
    log_event_mock = mocker.patch("app.slack.event_handler.log_event")
    req = FakeRequest({"event": {"type": "message"}}, user_id="U_X")
    await event_handler.log_event_middleware(req=req, resp=None, next=AsyncMock())
    log_event_mock.assert_not_called()  # 로깅 안 됨
    assert req.context["event"] == "message"  # context 는 여전히 설정
```

## 4. dependency_injection_middleware — 분기 6갈래

이 미들웨어는 5단계 분기를 거쳐요.

```python
event = req.context.get("event")
user_id = req.context.user_id

# 1. 우회 이벤트 (message, app_mention 등 6종)
if event in [...]:
    await next(); return

# 2. 등록 유저 → 의존성 주입
repo = SlackRepository()
user = repo.get_user(user_id)
if user:
    req.context["service"] = SlackService(...)
    req.context["point_service"] = PointService(...)
    req.context["user"] = user
    await next(); return

# 3. app_home_opened + 미등록 → None 주입 후 next
if event == "app_home_opened":
    req.context["service"] = None
    ...
    await next(); return

# 4. user_id=None (일부 슬랙 봇) → silent return
if user_id is None:
    return

# 5. 그 외 미등록 유저 → 관리자 채널 + BotException
await app.client.chat_postMessage(channel=ADMIN_CHANNEL, text=...)
raise BotException("사용자 정보를 찾을 수 없어요.")
```

각 분기를 별도 테스트로 박아두고, **5번 분기에서 BotException 이 raise 되기 전에 관리자 알림이 가는지**
까지 검증했어요.

```python
with pytest.raises(BotException) as exc_info:
    await event_handler.dependency_injection_middleware(...)
chat_post_mock.assert_awaited_once()
kwargs = chat_post_mock.await_args.kwargs
assert kwargs["channel"] == settings.ADMIN_CHANNEL
assert "U_GHOST" in kwargs["text"]
```

> 💡 **배운 것**: "예외를 던지기 전에 관리자에게 알림을 보낸다" 같은 부수효과 + 예외 동작은 한 테스트에서
> 함께 검증해야 안전. `pytest.raises` 컨텍스트 안에서 mock 호출도 함께 단언하면 둘이 같은 호출에서
> 일어났음을 보장할 수 있어요.

## 5. handle_error — 한국어 분기 정규식

에러 핸들러는 **한국어 정규식 (`[ㄱ-힝]`)** 으로 메시지에 한글이 있는지 체크해서, 한국어면
원본 메시지를 사용자에게 노출, 영문이면 "예기치 못한 오류" 로 마스킹해요.

```python
if re.search(r"[ㄱ-힝]", str(error)):
    message = str(error)  # 한국어 → 그대로 노출
else:
    message = "예기치 못한 오류가 발생했어요."  # 영문 → 마스킹
```

이는 "내부 디버깅용 메시지가 사용자에게 그대로 새는 걸 막는 안전망" 이에요. 한국어로 작성된 메시지는
이미 사용자 노출을 의도한 것이라 통과시키는 거죠. 영문/Pythonic 에러 메시지는 그대로 보여주면
오히려 혼란스러우니 일반화.

```python
# BotException("이미 인증한 글이에요.")  → 사용자에게 그대로
# RuntimeError("KeyError: 'foo'")        → "예기치 못한 오류가 발생했어요."
```

`ValueError` 는 별도 분기로 reraise 되고 사용자 알림이 안 가요. 단순 값 오류는 핸들러 자체가
잘못된 입력을 던졌다는 신호라 사용자에겐 노출하지 않는 정책이에요.
