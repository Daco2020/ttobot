# 015 - GitHub Actions CI 추가하기

테스트가 240개 통과하는 상태에서 **자동화**를 시작했어요. PR 마다 자동으로 테스트를 돌려주고
실패한 PR 은 머지를 막아주는 GitHub Actions CI 를 추가했습니다.

## 1. 무엇을 만들었나요?

세 가지 산출물이 추가됐어요.

| 파일                                | 역할                                                |
| ----------------------------------- | --------------------------------------------------- |
| `.env.test`                         | CI 와 격리 환경에서 쓸 더미 환경 변수                |
| `.github/workflows/test.yml`        | GitHub Actions 워크플로우 (uv sync + pytest)        |
| `test/conftest.py` 수정             | 외부 의존성 차단 + store/ 자동 보강 + 더미 RSA 키 처리 |

## 2. 왜 `.env.test` 가 필요했나요?

또봇의 `Settings` 는 슬랙 토큰, 구글 시트 URL, BigQuery 크리덴셜 등 **20+ 개 필드를 모두 require** 해요.
하나라도 빠지면 `import app.config` 자체가 깨져요. 테스트는 외부 호출을 모두 mock 으로 차단하니까
실제 값은 필요 없지만, **import 시점에 settings 검증을 통과** 해야 하긴 해요.

두 가지 선택지가 있었어요:

- **A. 코드를 고쳐서 settings 를 lazy-validation 으로 바꾼다** → 정책 위반 (코드 변경)
- **B. 더미 값으로 채운 `.env.test` 를 레포에 두고 CI 에서 복사 사용** → 코드 변경 없음 ✓

당연히 B 를 골랐어요.

## 3. `.env.test` 를 만들면서 마주친 함정 3가지

### 3-1. `BIGQUERY_CREDENTIALS` 의 `private_key` 는 PKCS#8 RSA 형식 검증

`app/bigquery/client.py` 가 모듈 로드 시점에:

```python
credentials = service_account.Credentials.from_service_account_info(
    info=settings.BIGQUERY_CREDENTIALS
)
```

이걸 호출해요. `from_service_account_info` 는 `private_key` 필드를 실제로 파싱해서 RSA 키인지 검증해요.
"x" 같은 placeholder 는 통과 못 합니다. `openssl genpkey -algorithm RSA` 로 진짜 PKCS#8 키를 더미로
생성해서 넣었어요.

### 3-2. `JSON_KEYFILE_DICT` 의 `private_key` 는 oauth2client 가 또 검증

같은 키를 `JSON_KEYFILE_DICT` 에도 넣어줘야 해요. `app/client.py` 의:

```python
credentials = ServiceAccountCredentials.from_json_keyfile_dict(
    settings.JSON_KEYFILE_DICT, settings.SCOPE
)
```

이 호출이 또 다른 검증을 해요. 같은 더미 키를 두 곳에 채우면 둘 다 통과해요.

### 3-3. `SpreadSheetClient` 가 default arg 에서 네트워크 호출 ⚠️

가장 까다로웠던 부분. `app/client.py` 의 클래스 정의:

```python
class SpreadSheetClient:
    _instance = None

    def __init__(
        self,
        doc: Spreadsheet = gc.open_by_url(settings.SPREAD_SHEETS_URL),  # ← 여기!
        sheets: dict[str, Worksheet] | None = None,
    ) -> None:
        ...
```

Python 의 default argument 는 **함수/클래스 정의 시점에 한 번 평가** 돼요. 즉 `app.client` 모듈을
import 하는 순간 `gc.open_by_url(...)` 이 호출되어 Google Sheets 로 **실제 네트워크 요청** 이 갑니다.
이 단계에서 OAuth refresh 가 일어나는데, 더미 크리덴셜로는 당연히 실패해요.

코드를 고치면 정말 쉬운 문제이지만 (lazy 평가로 바꾸기), 정책상 코드 변경은 금지. 대신 **conftest 의
최상단**에서 gspread 를 mock 으로 갈아끼웠어요.

```python
# test/conftest.py 최상단
import gspread
gspread.authorize = MagicMock(return_value=MagicMock())  # type: ignore[assignment]

# 이후에야 안전하게 app 모듈 import
from app.slack.repositories import SlackRepository  # noqa: E402
```

`gspread.authorize(credentials)` 가 MagicMock 을 반환하면, 그 MagicMock 의
`open_by_url(...)` 도 MagicMock 을 돌려주고, **네트워크 호출이 발생하지 않아요**. 코드 한 줄도
건드리지 않고 import 시점 네트워크 호출을 차단했어요.

## 4. `store/` 가 gitignored 라는 또 다른 함정

테스트가 통과하는 걸 격리된 디렉터리에서 검증해보니, **4개가 실패** 했어요. 모두 같은 패턴이었어요.

```
File "/.../app/models.py", line 29
    with open("store/writing_participation.csv", "r") as f:
FileNotFoundError
```

`User.is_writing_participation` property 가 진짜 CSV 를 직접 읽고 있어요. 로컬에는 있지만 (gitignored
지만 로컬에는 실데이터가 있음), **CI 의 깨끗한 체크아웃에는 없어요**.

해결: **세션 단위 autouse 픽스처**로 store/ 디렉터리에 **빈 CSV 헤더** 만 자동 생성. 이미 있는 파일은
건드리지 않으니 로컬 운영 데이터에는 영향 없음.

```python
@pytest.fixture(autouse=True, scope="session")
def _ensure_store_csv_skeleton() -> None:
    store_dir = Path("store")
    store_dir.mkdir(exist_ok=True)
    for name, header in _STORE_FILES.items():
        path = store_dir / name
        if path.exists():
            continue  # 로컬 운영 데이터 보호
        with path.open("w", newline="", encoding="utf-8") as f:
            writer = csv.writer(f, quoting=csv.QUOTE_ALL)
            writer.writerow(header)
```

이걸 넣고 다시 격리 검증을 했더니 240/240 통과.

## 5. GitHub Actions 워크플로우

본체는 짧아요.

```yaml
name: test
on:
  push:
    branches: [main]
  pull_request:

jobs:
  pytest:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true
      - name: Set up Python
        run: uv python install 3.11
      - name: Prepare test env
        run: cp .env.test .env
      - name: Sync dependencies
        run: uv sync --frozen
      - name: Run tests
        run: uv run pytest -q
```

핵심은 `--frozen` 플래그. `uv.lock` 파일에 박힌 정확한 버전을 그대로 설치해서, **로컬과 CI 가
같은 의존성 트리** 를 쓰게 강제해요. lock 이 없으면 통과하던 게 CI 에선 깨지는 흔한 사고를 방지.

`enable-cache: true` 는 uv 의존성을 GitHub Actions 캐시에 보존해요. 두 번째 실행부턴 `uv sync` 가
거의 즉시 끝나요.

## 8. 사용 안내

PR 을 올리면 자동으로 status check 가 보입니다. 빨갛게 뜨면 머지 전에 고치고, 초록이 뜨면 안심하고
머지. 향후 추가하기 좋은 단계는:

- **Branch protection rule** 설정 — main 으로 직접 push 금지, PR + 통과한 status check 필수
- **lint job 추가** — `uv run ruff check .` 를 별도 step 으로
- **mypy job 추가** — 점진적 도입 (정책상 코드 변경 필요 없는 영역만)
