# 006 - 미니 라우터 4종 테스트 작성하기 (포인트 / 메시지 / 인프런 / 글쓰기 참여)

투두 4번 섹션을 마무리했어요. 4개의 작은 라우터에 **총 18개 테스트** 가 추가됐어요.

## 1. 어떤 라우터인가요?

이번에 다룬 4개 라우터는 모두 "관리자/단일 목적" 의 작은 엔드포인트들이에요.

| 라우터                          | 메서드 | 권한      | 핵심 동작                          |
| ------------------------------- | ------ | --------- | ---------------------------------- |
| `/v1/points`                    | POST   | ADMIN     | 큐레이션/반상회/특별 보너스 포인트 지급 |
| `/v1/send-messages`             | POST   | ADMIN     | 슬랙 메시지 일괄 전송               |
| `/v1/inflearn/coupons`          | GET    | ADMIN     | 인프런 쿠폰 CSV 조회                |
| `/v1/writing-participation`     | GET    | (없음)    | 글쓰기 참여 신청 CSV 조회           |

각 라우터는 분기가 단순해서 라우터 단위 테스트의 **표준 패턴** 을 다지기 좋은 재료였어요.

## 2. `/v1/points` — 다중 분기 라우터의 모범 (8개)

이 라우터는 `point_type` 파라미터에 따라 호출하는 서비스 메서드가 달라져요.

```python
if point_type == PointTypeEnum.CURATION:
    for uid in user_ids:
        msg = point_service.grant_if_curation_selected(uid)
        await send_point_noti_message(...)
        await asyncio.sleep(1)
elif point_type == PointTypeEnum.VILLAGE_CONFERENCE: ...
elif point_type == PointTypeEnum.SPECIAL:
    if not point or not reason:
        raise HTTPException(400, ...)
    ...
```

### 2-1. 외부 의존성 4가지를 격리

```python
@pytest.fixture
def no_sleep(mocker):
    return mocker.patch("app.api.views.point.asyncio.sleep", new=AsyncMock())

@pytest.fixture
def send_noti_mock(mocker):
    return mocker.patch("app.api.views.point.send_point_noti_message", new=AsyncMock())
```

특히 `asyncio.sleep(1)` 봉쇄가 중요해요. 유저 N명이면 N초가 그대로 테스트 시간에 더해져요.
mock 으로 갈아끼우면 테스트는 즉시 끝나면서도 호출 카운트는 그대로 검증할 수 있어요.

### 2-2. 다룬 케이스

- ✅ curation: 유저별 service 호출 + 알림 메시지 전송 + text prefix 합쳐지는지
- 🌀 curation + 빈 user_ids: 200 + 호출 0회
- ✅ village_conference: 해당 service 메서드 호출
- ✅ special + point/reason: `grant_if_special_point(uid, point, reason)` 호출
- ⚠️ special + point=0: 400
- ⚠️ special + reason="": 400
- ⚠️ 비-admin: 403
- ⚠️ 인증 누락: 403

> 💡 **배운 것**: `point=0` 과 `reason=""` 은 **둘 다** 검증되어야 해요. 라우터의 `if not point or not reason` 은 OR 조건이라
> 둘 중 하나만 빠져도 400. 두 케이스를 따로 테스트해야 OR 의 양쪽 가지가 모두 살아있는지 확인할 수 있어요.

## 3. `/v1/send-messages` — 단순 위임 라우터 (4개)

권한 체크 → for 루프로 슬랙 메시지 전송. 분기 없음.

- ✅ admin → 각 항목으로 chat_postMessage 호출. `await_args_list` 로 호출 순서까지 검증.
- ⚠️ 비-admin → 403
- ⚠️ 인증 누락 → 403
- 🌀 빈 리스트 → 200 + 호출 0회

## 4. `/v1/inflearn/coupons` — CSV 읽기 라우터 (4개)

가장 단순. CSV 파일을 그대로 dict 리스트로 반환.

- ✅ admin + 데이터 있음 → 그대로 반환
- ⚠️ 비-admin → 403
- ⚠️ 인증 누락 → 403
- 🌀 헤더만 있는 CSV → 빈 리스트

`tmp_store` 픽스처가 임시 디렉터리로 cwd 를 옮겨주니, 진짜 인프런 쿠폰 데이터를 건드릴 일이 없어요.

## 5. `/v1/writing-participation` — 인증 없는 CSV 라우터 (2개)

이 라우터는 **인증이 없어요.** 누구나 글쓰기 참여 신청 목록을 볼 수 있는 구조라 테스트도 단순해요.

- ✅ 행이 있을 때 dict 리스트로 반환
- 🌀 헤더만 있는 CSV → 빈 리스트

> 💡 **운영 관점에서 살짝 의심스러운 부분**: 글쓰기 참여 명단이 인증 없이 노출되는 게 의도된 것인지는
> 한 번 확인이 필요해요. 다만 이번 작업은 테스트 보강 범위라 코드는 손대지 않았어요.
