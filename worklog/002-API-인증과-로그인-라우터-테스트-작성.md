# 002 - API 인증과 로그인 라우터 테스트 작성하기

투두 문서의 1번 섹션을 마무리했어요. `app/api/auth.py` 의 토큰 함수와 `current_user` 의존성,
그리고 `app/api/views/login.py` 의 4개 엔드포인트(`/slack/login`, `/slack/auth`, `/slack/auth/refresh`, `/slack/me`)에
**총 23개의 테스트** 를 추가했습니다.

## 1. 왜 인증부터 시작했나요?

테스트 투두에서 1번에 둔 데에는 이유가 있어요.

- 인증은 **거의 모든 다른 라우터의 전제조건** 이에요. 종이비행기, 메시지 전송, 포인트 지급 — 전부
  `current_user` 의존성을 통해야 들어갈 수 있어요.
- 인증을 먼저 완전히 검증해두면, 다른 라우터 테스트에서는 "이 케이스는 인증 단에서 막힌다" 라고
  믿고 넘어갈 수 있어요.
- 토큰 함수는 외부 의존성이 거의 없는 **순수 함수** 라서 테스트하기 가장 쉬워요. 인프라가 제대로
  깔렸는지 검증하기에도 좋아요.

## 2. 어떤 케이스를 다뤘나요?

### 2-1. `auth.py` 단위 테스트 (12개)

`encode_token` / `decode_token` — JWT 인코딩과 디코딩.

- ✅ round-trip: 인코드 → 디코드 후 `user_id`, `iss`, `iat`, `exp` 보존
- 🌀 한글/이모지/특수문자가 들어가도 보존 ("김은찬", "🚀 테스트 #1")
- ⚠️ 만료된 토큰 → `jwt.ExpiredSignatureError`
- ⚠️ 잘못된 서명 → `jwt.InvalidSignatureError`
- ⚠️ 잘못된 issuer → `jwt.InvalidIssuerError`

`current_user` 의존성 — 받은 토큰으로 유저를 찾아 `SimpleUser` 를 만들어 돌려줘요.

- ✅ 정상 토큰 + 유저 존재 → `SimpleUser` 반환
- ⚠️ Authorization 헤더 없음 → 403
- ⚠️ 망가진 토큰 → 403
- ⚠️ refresh 타입 토큰을 access 로 쓰면 → 403
- ⚠️ 토큰은 유효하지만 유저 없음 → 404
- 🌀 user_id 가 빈 문자열 → repository 조회조차 안 하고 404

`current_user` 는 직접 비동기 함수로 호출해 테스트했어요. `HTTPAuthorizationCredentials` 객체를 손으로
만들고 `MagicMock()` 으로 repo 를 흉내냈죠. TestClient 로 endpoint 를 거쳐서 테스트하는 것보다 훨씬 더
빠르고 부수효과도 없어요.

### 2-2. `login.py` 라우터 테스트 (11개)

`/v1/slack/login` (1개)

- ✅ state 발급 후 redirect_url 반환. `oauth_flow.issue_new_state` 와 `authorize_url_generator.generate` 를
  mock으로 갈아끼웠어요.

`/v1/slack/auth` (4개)

- ✅ 정상 code → access/refresh 토큰 JSON 응답. 이때 access 에는 `type` 키가 없고 refresh 에는
  `type=refresh` 가 있는지까지 확인.
- ⚠️ slack 이 `error` 쿼리를 돌려보낸 경우 → 404
- ⚠️ code 가 없으면 → 403
- ⚠️ `run_installation` 이 None 반환 → 403

`/v1/slack/auth/refresh` (4개) — **여기서 버그를 발견했어요.**

`/v1/slack/me` (2개)

- ✅ 인증된 유저 → SimpleUser JSON
- ⚠️ 인증 누락 → 403

## 3. 발견한 버그: `/slack/auth/refresh` 의 잘못된 분기

라우터 코드는 이렇게 되어 있어요.

```python
@router.get("/slack/auth/refresh")
async def slack_auth_refresh(refresh_token, service):
    try:
        decoded_payload = decode_token(refresh_token)
        if decoded_payload.get("type") != "refresh":
            return HTTPException(status_code=403, detail="...")  # ← return!
        user = service.get_user_by(user_id=decoded_payload["user_id"])
        if not user:
            return HTTPException(status_code=404, detail="...")  # ← return!
    except PyJWTError:
        return HTTPException(status_code=403, detail="...")  # ← return!
    ...
```

`raise HTTPException(...)` 이 아니라 `return HTTPException(...)` 으로 되어 있어요. FastAPI 는 이걸
"예외" 가 아닌 **"라우터의 응답값"** 으로 받아서 직렬화해요. 결과적으로:

- 클라이언트는 `200 OK` 를 받아요.
- 본문은 HTTPException 객체가 직렬화된 이상한 모양으로 떨어져요.
- access_token 은 발급되지 않으니 클라이언트는 결국 로그인을 다시 해야 해요.

이건 명백한 버그지만, **이번 작업의 범위는 "테스트 보강"** 이라 코드 수정은 미뤘어요. 대신
테스트는 **현재 동작을 그대로 기록** 하도록 작성했어요. 함수 이름과 docstring 에 `_buggy_200` 을
넣어서, 나중에 코드를 고치면 이 테스트도 같이 갱신해야 한다는 신호를 남겼습니다.

```python
def test_slack_auth_refresh_with_access_typed_token_returns_buggy_200(...):
    """⚠️ access 타입 토큰을 refresh 로 사용 → 현재 구현은 200 + HTTPException 직렬화 본문.

    `if ...: return HTTPException(...)` 이 raise 가 아닌 return 으로 작성되어 있어 발생하는 동작.
    """
```

이렇게 해두면 나중에 누군가 라우터를 고쳤을 때 이 테스트들이 깨지면서 "이 버그는 의도된 것이 아니다"
라는 사실이 자연스럽게 드러나요.

## 4. 테스트를 작성하면서 마주친 작은 트릭들

### 4-1. `auth_for(user)` — 의존성 오버라이드

`/v1/slack/me` 같은 인증 필요 라우터를 테스트할 때, 매번 진짜 JWT 를 발급하는 건 번거로워요.
`test/api/conftest.py` 에 만들어둔 `auth_for` 픽스처가 이 문제를 해결해요.

```python
def test_slack_me_returns_current_user(client, auth_for, factory):
    user = factory.make_user(user_id="U_ME", name="나야나")
    auth_for(user)
    response = client.get("/v1/slack/me")  # 헤더 없이도 인증된 척
```

내부적으로는 `app.dependency_overrides[current_user] = lambda: simple_user` 를 세팅해요.
테스트가 끝나면 `client` 픽스처 finalizer 가 `dependency_overrides.clear()` 를 호출해서
다음 테스트로 새지 않게 합니다.

### 4-2. `override_api_service` — 서비스 갈아끼우기

`/slack/auth/refresh` 는 `service.get_user_by` 를 호출해요. 진짜 CSV 를 읽으면 테스트 격리가 깨지니
`MagicMock` 으로 만든 가짜 서비스를 의존성으로 주입했어요.

```python
service = MagicMock()
service.get_user_by.return_value = user
override_api_service(service)
```

이 패턴은 앞으로 종이비행기, 포인트, 메시지 라우터를 테스트할 때도 쭉 재활용돼요.

### 4-3. `mocker.patch.object(login_view.oauth_flow, "run_installation", ...)`

`oauth_flow` 는 모듈 로드 시점에 만들어지는 싱글톤이에요. 테스트마다 매번 내부 메서드를 갈아끼우는
방식으로 처리했어요. 모듈 import 자체는 건드리지 않고, 인스턴스 메서드만 mock 해서 OAuth 외부 호출이
일어나지 않도록 막았습니다.
