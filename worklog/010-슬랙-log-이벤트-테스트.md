## 2. 가장 까다로웠던 부분 — `handle_reaction_added`

이 핸들러는 한 번 호출되면 **3단계 분기 결정** 을 해요.

```python
# 1단계: 항상 emoji 큐에 적재
emojis_upload_queue.append(...)

# 2단계: 공지 채널 + noti-check 인가?
if channel == NOTICE_CHANNEL and reaction == "noti-check":
    if await _is_thread_message(...):  return         # 스레드 메시지면 종료
    if _is_checked_notice(...):         return        # 이미 확인했으면 종료
    if 3일_초과:                        return        # 너무 오래된 공지면 종료
    point_service.grant_if_notice_emoji_checked(...)  # 포인트 지급
    _write_checked_notice(...)                        # 기록 저장

# 3단계: PRIMARY 채널 + catch-kyle + super_admin 글 인가?
if channel in PRIMARY_CHANNEL and reaction == "catch-kyle" and item_user == TTOBOT_USER_ID:
    if _is_checked_super_admin_post(...):  return
    if 1일_초과:                          return
    content = repo.get_content_by(ts=...)
    if content is None or content.user_id != SUPER_ADMIN:  return
    point_service.grant_if_super_admin_post_reacted(...)
    _write_checked_super_admin_post(...)
```

분기마다 다른 외부 의존성을 부르니까, 테스트는 **한 분기에 하나의 negative path** 를 박아두는 식으로
9개를 작성했어요.

| 케이스                                   | 분류  | 결과                                |
| ---------------------------------------- | ----- | ----------------------------------- |
| 일반 채널 + 일반 reaction                | ✅    | emoji 큐만 적재                     |
| 공지 + noti-check + 신규 + 3일 이내      | ✅    | 포인트 지급 + 기록 저장             |
| 공지 + 스레드 메시지                     | ⚠️   | 포인트 지급 X                       |
| 공지 + 이미 확인                         | ⚠️   | 포인트 지급 X                       |
| 공지 + 3일 초과                          | ⚠️   | 포인트 지급 X                       |
| PRIMARY + catch-kyle + 신규 + 1일 이내   | ✅    | 성윤 포인트 지급                    |
| PRIMARY + catch-kyle + 1일 초과          | ⚠️   | 포인트 지급 X                       |
| PRIMARY + catch-kyle + 작성자 != super   | ⚠️   | 포인트 지급 X                       |
| PRIMARY + catch-kyle + 이미 확인         | ⚠️   | 포인트 지급 X                       |

## 3. `aiocache.cached` 데코레이터의 함정

`_is_thread_message` 는 `@cached(ttl=60)` 으로 감싸여 있어요. Slack API rate limit 을 피하려는 캐싱.

```python
@cached(ttl=60, key_builder=_is_thread_message_cache_key_builder)
async def _is_thread_message(client, channel_id, ts):
    res = await client.conversations_replies(channel=channel_id, ts=ts)
    ...
```

테스트에서 이걸 그대로 쓰면 **이전 테스트의 결과가 다음 테스트로 새요**. 첫 번째 테스트에서 `(C_X, ts_1) → False` 캐시되면, 두 번째 테스트에서 같은 인자로 부르면 진짜 코드는 안 돌고 캐시 값만 돌아와요.

**해법**: 각 테스트마다 **다른 channel_id / ts** 를 사용해서 캐시 키를 매번 다르게 만들었어요.

```python
# test 1
ts="1.0", channel_id="C_T1"

# test 2
ts="2.0", channel_id="C_T2"

# test 3
ts="3.0", channel_id="C_T3"
```

이러면 캐시가 깨지지 않아도 각 테스트가 진짜 mock 함수를 부르게 돼요. 테스트 간 상태 격리가 쉬운 방식.

## 4. CSV 기반 멱등성 — `_is_checked_*` / `_write_checked_*`

`_write_checked_notice` 는 처음 호출 시 헤더를 쓰고, 이후엔 행만 추가해요. `_is_checked_notice` 는 같은
파일을 읽어서 (user_id, notice_ts) 튜플이 있는지 확인.

`tmp_store` 픽스처가 cwd 를 임시 디렉터리로 옮겨주니, 진짜 `store/_checked_notice.csv` 를 건드리지
않아요. 그 다음 두 패턴으로 검증:

```python
# 신규 기록 → 다음 호출에서 True
log_events._write_checked_notice("U_X", "1.0")
assert log_events._is_checked_notice("U_X", "1.0") is True

# 다른 user_id → False
log_events._write_checked_notice("U_A", "1.0")
assert log_events._is_checked_notice("U_B", "1.0") is False
```

> 💡 **배운 것**: 멱등성을 보장하는 함수는 "쓰고 → 다시 읽었을 때 진짜 들어가있는지" 가 핵심.
> 한 함수에 두 동작을 합쳐서 테스트하면 round-trip 이 자연스럽게 검증돼요.

## 5. BigQuery 큐 mock — 모듈 전역 변수 갈아끼우기

`bigquery_queue.comments_upload_queue` 는 **모듈 전역 리스트** 라 그냥 import 해서 append 하면 진짜
큐가 더러워져요. `mocker.patch` 로 빈 리스트로 갈아끼우면 깔끔하게 격리돼요.

```python
queue = mocker.patch(
    "app.slack.events.log.bigquery_queue.comments_upload_queue", new=[]
)

await log_events.handle_comment_data(body=body)

assert len(queue) == 1
assert queue[0]["user_id"] == "U_X"
```

`new=[]` 가 핵심. mock 객체가 아니라 진짜 빈 리스트로 대체하기 때문에 핸들러가 `.append()` 를 부르면
리스트에 정상적으로 들어가요.
