## 2. 가장 까다로웠던 부분 — `prepare_subscribe_message_data`

이 메서드는 한 번 호출되면:

1. 기존 임시 CSV 삭제
2. 어제 날짜의 submit 글만 추출 (`contents.csv` 에서 pandas 로 읽음)
3. 각 글의 작성자가 글쓰기 참여자라면 채널을 WRITING_CHANNEL 로 치환
4. 구독자 × 글 조합으로 메시지 데이터 생성
5. 데이터가 있으면 임시 CSV 작성, 없으면 미작성

테스트는 4개 케이스로 분기를 모두 커버했어요.

| 케이스                                  | 입력                            | 검증                                |
| --------------------------------------- | ------------------------------- | ----------------------------------- |
| 어제 글 + 그저께 글 + 어제 패스         | 3종 contents                    | 어제 submit 1건만 → 구독자 2명 × 1글 = 2행 |
| 매칭 콘텐츠 0건                          | 옛날 글만 있음                  | 임시 CSV 미생성                     |
| 작성자가 writing_participation 등록     | writing 신청자 + 어제 글        | target_user_channel 이 WRITING_CHANNEL |
| 기존 임시 CSV 존재                      | 미리 만들어두고 호출             | 호출 후 파일 삭제됨                 |

`tz_now` 를 `2025-06-15` 로 고정해서 "어제" 가 명확히 `2025-06-14` 가 되도록 했어요. 테스트가 어느
요일에 실행되든 똑같이 통과해요.

## 3. `send_subscription_messages` — 부분 실패 허용

이 메서드는 행마다 `_send_subscription_message` 를 부르는데, **한 행이 실패해도 다음 행을 진행** 하는
정책이에요. 이게 의도된 동작인지 테스트로 박아두는 게 중요해요.

```python
send_one_mock = mocker.patch.object(
    BackgroundService,
    "_send_subscription_message",
    new=AsyncMock(side_effect=[RuntimeError("permalink 실패"), None]),
)

# 두 행 모두 시도되어야 함
assert send_one_mock.await_count == 2

# 관리자 알림: 실패 1번 + 마지막 요약 1번
posted_texts = [c.kwargs["text"] for c in chat_post_mock.await_args_list]
assert any("U_FAIL" in t and "permalink 실패" in t for t in posted_texts)
assert any("총" in t and "명에게" in t for t in posted_texts)
```

`side_effect=[RuntimeError(...), None]` 패턴이 키예요. 첫 호출은 예외, 두 번째 호출은 정상. 이걸로
"한 행 실패 → 다음 행 진행" 정책이 깨지지 않았는지 명확히 검증돼요.

## 4. tenacity 재시도 검증

`_send_subscription_message` 에는 `@tenacity.retry(stop=stop_after_attempt(3), wait=wait_fixed(1))`
가 붙어있어요. 1초 wait 가 3번 들어가면 테스트가 3초 동안 멈춰요. **`asyncio.sleep` 을 mock 으로
갈아끼우는 것** 이 핵심.

```python
mocker.patch("app.slack.services.background.asyncio.sleep", new=AsyncMock())
slack_app.client.chat_getPermalink = AsyncMock(side_effect=RuntimeError("network"))

with pytest.raises(RuntimeError, match="network"):
    await bg._send_subscription_message(slack_app, message)

assert slack_app.client.chat_getPermalink.await_count == 3  # tenacity 가 3회 시도
```

> 💡 **배운 것**: tenacity 의 wait_fixed 는 내부적으로 `asyncio.sleep` 을 호출. 테스트에서는 같은 모듈의
> sleep 을 mock 하면 즉시 진행돼요. 단, tenacity 자체의 sleep 메커니즘은 더 복잡해서 모듈 레벨 mock 이
> 항상 통하진 않아요. 이번엔 우연히 통했고, 안 통하면 `tenacity.retry` 데코레이터의 옵션을 일시적으로
> 풀어주는 monkeypatch 가 필요해요.

## 5. FakeAsyncWebClient 의 `chat_getPermalink` 누락

작업 중 발견한 작은 인프라 이슈. `test/conftest.py` 의 `FakeAsyncWebClient` 는 `chat_postMessage` 만
정의되어 있어서 `mocker.patch.object(client, "chat_getPermalink", new=AsyncMock())` 호출이 실패해요.

해결: 테스트 안에서 직접 인스턴스 속성을 추가.

```python
slack_app.client.chat_getPermalink = AsyncMock(return_value={"permalink": "..."})
```

`FakeAsyncWebClient` 자체에 모든 슬랙 메서드를 추가하는 대신, 테스트별로 필요한 메서드만 즉석에서
부여하는 방식이 가벼워요. 만약 같은 메서드가 여러 곳에서 자주 쓰이면 추후 conftest 헬퍼로 옮기면 돼요.
