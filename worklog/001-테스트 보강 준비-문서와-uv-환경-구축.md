# 001 - 테스트 보강을 시작하기 전, 문서와 uv 환경 구축하기

또봇은 백엔드 API 서버이자 슬랙 봇이에요. 두 얼굴 모두 동작하지만, 테스트는 일부 비즈니스 로직(포인트, 리마인더)에만 있고
API 라우터와 슬랙 이벤트 핸들러에는 거의 없는 상태였어요. 그래서 "성공/실패/엣지 케이스를 빠짐없이 작성해서 견고하게 만들기" 라는 목표를 세웠고,
그 첫걸음으로 **계획서**, **투두 문서**, **uv 가상환경**, **테스트 인프라** 까지 준비하는 작업을 했습니다.

이번 글은 코드를 한 줄도 짜기 전에 한 일들을 차근차근 풀어쓴 기록이에요.

---

## 1. 왜 이 작업이 필요했나요?

테스트가 거의 없으면 다음 두 가지가 어려워요.

1. **리팩터링이 무서워져요.** 잘 돌던 기능이 망가져도 알 수가 없어요.
2. **새 기능을 추가할 때 회귀를 못 막아요.** 종이비행기 코드를 고쳤는데 글 제출 흐름이 깨질 수 있어요.

또봇은 슬랙 API, 구글 시트, BigQuery 같은 외부 의존성이 많아서 "그냥 다 통합 테스트로 돌리자" 가 안 통해요.
**외부 의존성은 mock으로 대체**하고, 우리가 작성한 코드의 분기점만 골라 테스트하는 전략이 필요했죠.

그러려면 시작하기 전에 두 가지가 정해져 있어야 했어요.

- 어떤 함수에 어떤 케이스를 테스트할지의 **체크리스트**
- 테스트가 빠르고 격리되게 돌 수 있는 **개발 환경**

## 2. 어떤 문제가 있었나요?

이 프로젝트의 출발선은 다음과 같았어요.

- `requirements.txt`에 100여 개의 패키지가 정확한 버전으로 박혀 있었어요. `pip install -r requirements.txt` 가 잘 되면 다행이지만,
  파이썬 버전이 달라지거나 빌드 휠이 없으면 바로 깨졌어요.
- 테스트 디렉터리는 `test/test_point.py`, `test/test_reminder.py` 두 파일뿐. 늘어날 테스트를 어디에 어떻게 둘지 정해진 게 없었어요.
- 슬랙 이벤트 핸들러는 함수 인자로 `ack`, `body`, `client` 같은 슬랙-볼트 객체를 받아요. 직접 호출해 테스트하려면
  이 객체들의 **가짜 버전**을 매번 손으로 만들어야 했어요.
- FastAPI 테스트도 마찬가지로 `TestClient`, JWT 토큰 발급, 의존성 오버라이드 같은 **공통 작업** 이 매 테스트마다 반복될 게 뻔했어요.

## 3. 어떻게 해결했나요?

### 3-1. 먼저 문서부터 두 장

테스트를 쓰기 전에 "무엇을, 어느 순서로 쓸지" 를 정리하지 않으면 중간에 길을 잃어요. 그래서 두 문서를 먼저 만들었어요.

- `docs/01-refactoring-plan.md` — 프로젝트 구조 요약, 외부 의존성 지도, API/슬랙 핸들러 표,
  uv 전환 계획, 단계별 작업 흐름까지 모두 담은 **계획서**
- `docs/02-test-todo.md` — 13개 섹션, 약 180개 체크박스의 **투두 문서**.
  각 항목은 ✅ 성공 / ⚠️ 실패 / 🌀 엣지 셋 중 하나로 분류했어요.

이렇게 해두니 작업이 막히거나 중간에 사람이 바뀌어도, 어디서부터 이어가면 되는지 한눈에 보여요.

### 3-2. pip → uv 전환

`uv`는 Rust로 짜인 차세대 파이썬 패키지 매니저예요. `pip + venv` 보다 훨씬 빠르고, 락 파일(`uv.lock`)을 자동으로 관리해줘요.

- `pyproject.toml`을 새로 작성했어요. 런타임 의존성은 `[project].dependencies`에, 테스트 도구(pytest, ruff 등) 는
  `[dependency-groups].dev` 로 분리했어요. 분리해두면 운영 서버에 굳이 ruff까지 깔리지 않아요.
- `uv sync` 한 번이면 `.venv` 와 `uv.lock` 이 동시에 생성돼요.
- `Makefile`은 `pip install` 대신 `uv sync`를, `uvicorn ...` 대신 `uv run uvicorn ...`을 쓰도록 바꿨어요.
  `make test`, `make lint` 타겟도 추가했어요.

#### 중간에 만난 작은 모험들

전환 과정에서 의존성 충돌 세 개를 만났어요.

1. **`oauth2client` 누락** — 옛 requirements.txt 에는 있었는데 옮길 때 빠뜨렸어요. 추가하면 끝.
2. **`aiohttp` 신버전이 import 시 이벤트 루프를 요구** — 슬랙-볼트의 소켓 모드 핸들러가 모듈 로드 시점에
   `ClientSession`을 만들어서 `RuntimeError: no running event loop`가 났어요. 코드 변경 없이는 못 고쳐서
   `aiohttp<3.10` 으로 버전을 묶었어요.
3. **`googletrans` 가 httpx 옛 버전을 강제 고정** — `googletrans==4.0.0rc1`이 `httpx==0.13.3`에 못박혀 있었는데,
   FastAPI TestClient는 `httpx>=0.24` 가 필요했어요. 알고 보니 `googletrans 4.0.2` 가 이 문제를 이미 고쳐서
   배포되어 있었어요. 4.0.2 로 올리니 끝.

> 💡 **배운 것**: requirements.txt를 그대로 옮기지 말고, 한 번 정도는 의존성을 다듬을 기회를 가지자. 락 파일이 있으니 재현은 어차피 보장돼요.

### 3-3. 테스트 디렉터리 재정비

원래 한 폴더에 흩어져 있던 테스트들을 목적별로 나눴어요.

```
test/
├── conftest.py            # 모든 테스트가 공유하는 픽스처
├── factories.py           # User, Content 같은 도메인 모델 빌더
├── api/                   # FastAPI 라우터 테스트가 들어갈 곳
│   ├── conftest.py        # TestClient, JWT, 의존성 오버라이드
│   └── test_smoke.py      # 인프라가 살아있는지 확인하는 짧은 테스트
├── slack/                 # 슬랙 이벤트 핸들러 테스트가 들어갈 곳
│   ├── conftest.py        # ack/say mock + 슬랙 body 빌더
│   └── test_smoke.py      # 핸들러 직접 호출 가능한지 확인
└── services/              # 비즈니스 로직 (포인트/배경 작업) 테스트
    ├── test_point_service.py
    └── test_background_service.py
```

핵심은 **"테스트 작성자는 픽스처만 받아서 쓰면 된다"** 라는 원칙이에요.

#### `factories.py` — 모델 팩토리

```python
def make_user(**overrides):
    defaults = {"user_id": "U_TEST", "name": "테스트유저", ...}
    defaults.update(overrides)
    return models.User(**defaults)
```

이런 함수들이 모여 있어요. 테스트에서는 `factory.make_user(name="덕순")` 처럼 필요한 필드만 덮어쓰면 끝.

#### `test/api/conftest.py` — API 테스트의 든든한 친구들

- `client` — FastAPI `TestClient`. **`with` 블록으로 감싸지 않아서** 슬랙 소켓 모드 핸들러의 startup이 안 발화해요.
- `auth_for(user)` — `current_user` 의존성을 그 유저로 고정. 진짜 토큰을 안 만들어도 인증된 것처럼 동작해요.
- `make_access_token`, `make_refresh_token` — 진짜 JWT가 필요할 때 사용.
- `override_api_repo`, `override_api_service`, `override_point_service` — 라우터가 의존하는 서비스를 mock으로 갈아끼우는 헬퍼.

#### `test/slack/conftest.py` — 슬랙 핸들러 테스트의 보일러플레이트 제거기

슬랙은 들어오는 페이로드(body)가 종류별로 형태가 달라요. 명령어, 액션, 뷰 제출, 메시지, 리액션 — 다섯 가지를
**빌더 함수** 로 만들어 두었어요. 테스트에서는 `slack_bodies.command(user_id="U_X")` 처럼 한 줄로 만들 수 있어요.

또 `fake_slack_client` 픽스처는 `AsyncMock` 한 덩어리. `chat_postMessage`, `views_open`, `conversations_history` 등
어떤 메서드든 자동으로 mock이 만들어지고, 호출됐는지/어떤 인자로 호출됐는지를 그대로 검증할 수 있어요.

#### `tmp_store` 픽스처 — CSV 격리

또봇은 `store/*.csv` 를 직접 읽고 써요. 테스트에서 진짜 파일을 건드리면 데이터가 꼬여요. 그래서:

```python
def test_xxx(tmp_store, csv_writer_helper):
    csv_writer_helper(tmp_store / "users.csv", [...], [{...}])
    # 코드가 "store/users.csv" 를 읽으면 임시 디렉터리 쪽을 본다
```

`monkeypatch.chdir(tmp_path)` 으로 작업 디렉터리를 임시 폴더로 옮기는 트릭이에요.

### 3-4. 마지막에 스모크 테스트 두 줄

인프라가 다 갖춰져도, 정말로 동작하는지는 한번 돌려봐야 알아요. 그래서:

- **API 쪽**: `GET /` 가 200 을 주고, 토큰 없이 `/v1/slack/me` 를 호출하면 403 이고, `auth_for` 로
  유저를 고정하면 그 유저 정보가 돌아오는지 — 이렇게 세 가지 케이스로 인프라를 점검했어요.
- **슬랙 쪽**: `handle_app_mention` 이 `ack` 외에 다른 호출을 하지 않는지, `open_help_view` 가
  `views_open` 을 정확히 한 번 호출하는지를 확인했어요.

```
$ uv run pytest test/
========================== 10 passed in 2.03s ==========================
```

기존 5개 테스트도 그대로 통과, 신규 5개 스모크도 통과. 안심하고 다음 단계로 갈 수 있게 됐어요.

## 5. 다음 단계

이제 본격적으로 테스트를 쓸 차례예요. 투두 문서의 **1번 섹션 (API 인증 & 로그인)** 부터 시작해서
`auth.py` 의 토큰 인코딩/디코딩, `current_user` 의존성, OAuth 콜백 라우터까지 차례대로 채워나갈 예정이에요.

이 단계를 건너뛰고 바로 테스트를 썼다면, 매 테스트마다 픽스처를 손으로 만들면서 시간이 줄줄 새고
일관성도 깨졌을 거예요. 시작하기 전에 한 박자 멈춰서 길을 깔아두니, 앞으로의 작업은 그 위를 달리기만 하면 돼요.
